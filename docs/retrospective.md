# PromptBuilder Retrospective

이 문서는 PromptBuilder가 처음의 깨진 상태에서 현재 구조로 오기까지의 고민과 결정을 정리한 회고입니다.

## 1. 처음에 보였던 문제

초기에는 화면과 문서, 그리고 일부 코드가 한 번에 너무 많이 섞여 있었습니다.

- 깨진 문자열 때문에 빌드가 자주 실패했습니다.
- 기능이 커질수록 파일 간 책임 경계가 흐려졌습니다.
- 홈 화면은 정보가 많았지만, 사용자 입장에서는 무엇부터 해야 하는지 덜 분명했습니다.
- 히스토리는 아래로 길게 내려가야 이전 기록을 볼 수 있었습니다.
- 관리자 화면은 외부 예시 문구와 프로젝트 문구가 섞여 있었습니다.

여기서 가장 먼저 들었던 고민은 "기능을 더 넣는 것보다, 지금 보이는 것을 먼저 정리해야 하지 않나?"였습니다.

## 2. 첫 번째 전환: 인코딩과 경로 문제 정리

가장 먼저 해결한 문제는 빌드 실패였습니다.

- `prompt-services.ts`의 문자열 깨짐 때문에 Vite/esbuild가 파싱하지 못했습니다.
- 파일 분리 이후에는 export/import 경계가 맞지 않아 다시 오류가 생겼습니다.
- `index.tsx`가 bootstrap 경로를 제대로 참조하지 못한 적도 있었습니다.

이 과정에서 얻은 결론은 명확했습니다.

- 코드가 복잡해질수록 문자열과 경로가 가장 먼저 무너집니다.
- 그래서 기능보다 먼저 구조와 인코딩을 정리해야 합니다.

## 3. 두 번째 전환: 기능 분리와 역할 정리

다음에는 기능을 파일별로 나눴습니다.

- 설정은 config 쪽으로 보냈습니다.
- 필드 생성은 fields 쪽으로 보냈습니다.
- 품질 분석은 quality 쪽으로 모았습니다.
- helpers.ts는 연결 계층으로 줄였습니다.

이때의 핵심 판단은 "분리는 필요하지만, 분리가 곧 중복이어서는 안 된다"였습니다.

즉, 파일을 나누되 책임 경계를 명확하게 두고, 서로를 다시 복제하지 않도록 정리해야 했습니다.

## 4. 세 번째 전환: UI는 기능보다 우선순위가 먼저다

화면을 보며 가장 크게 느낀 문제는 여백과 스크롤이었습니다.

- 글자 수에 비해 칸이 너무 넓었습니다.
- 한 화면에서 봐야 할 요소가 너무 세로로 쌓였습니다.
- 버튼과 통계, 모드 선택이 서로의 흐름을 끊었습니다.

그래서 한 번에 모든 것을 보여주기보다, 사용자 시선을 기준으로 다시 배치했습니다.

- 모드 선택과 빠른 흐름을 한 덩어리로 묶었습니다.
- AI 스타일 패널은 오른쪽 높이를 맞췄습니다.
- 히어로 문구는 너무 튀지 않도록 줄 간격과 폭을 다시 맞췄습니다.

결론은 "보기 좋게 비우는 것"보다 "불필요한 빈 공간을 없애는 것"이었습니다.

## 5. 네 번째 전환: 히스토리는 아래가 아니라 옆으로 봐야 한다

히스토리 개선은 아주 직접적인 사용자 피드백에서 시작됐습니다.

- 이전 버전을 아래로 스크롤해서 찾는 방식은 불편했습니다.
- 사용자는 기록을 순서대로 훑기보다, 특정 버전을 바로 골라 보고 싶어 했습니다.

그래서 히스토리는 스크롤형에서 탭형으로 바뀌었습니다.

- 선택 목록과 상세 패널을 분리했습니다.
- 버전 탭으로 이전 기록을 바로 전환하게 했습니다.
- 불러오기와 복사도 선택된 버전을 기준으로 동작하게 했습니다.

이건 단순한 레이아웃 수정이 아니라, 탐색 방식 자체를 바꾼 결정이었습니다.

## 6. 다섯 번째 전환: 프롬프트는 레이어로 나눠야 한다

프롬프트 생성에서는 특히 중요한 깨달음이 있었습니다.

현재 요청은 사실 세 종류가 섞여 있었습니다.

- System: 스타일과 가드레일
- Template: 구조
- User Input: 실제 입력값

이 셋을 한 덩어리로 섞으면 토큰이 불필요하게 늘고, 재사용성이 떨어졌습니다.

그래서 복잡도에 따라 두 갈래로 나눴습니다.

- 낮은 복잡도: 간단 템플릿
- 높은 복잡도: 구조화된 템플릿

이 선택은 기능을 줄이자는 뜻이 아니었습니다.  
오히려 상황에 따라 필요한 것만 꺼내 쓰게 만드는 쪽에 가깝습니다.

## 7. 여섯 번째 전환: 문서는 기록이 아니라 운영 도구다

작업이 누적되면서 문서의 역할도 바뀌었습니다.

- `README.md`
- `docs/phase.md`
- `docs/plan.md`
- `docs/retrospective.md`
- `webapp/public/static/features/common/changelog.js`

이 파일들은 단순 설명이 아니라, 다음 작업자가 현재 상태를 이해하는 기준이 되었습니다.

그래서 새 기능을 넣을 때마다 다음을 함께 갱신하는 규칙을 만들었습니다.

- 무엇이 바뀌었는지
- 왜 그렇게 바꿨는지
- 다음 우선순위가 무엇인지
- 현재 어떤 phase에 속하는지

## 8. 지금까지의 결론

지금의 PromptBuilder는 처음보다 훨씬 안정적입니다.

- 빌드와 테스트가 반복적으로 통과합니다.
- 프롬프트 생성과 최적화 흐름이 분리되어 있습니다.
- 히스토리와 관리자 화면은 더 읽기 쉽게 정리됐습니다.
- 문서와 changelog가 현재 구현을 따라가고 있습니다.

하지만 아직 끝난 것은 아닙니다.

남은 과제는 다음과 같습니다.

- 결과 3개 생성과 비교 UX를 더 안정화하기
- Optimize 흐름을 더 짧게 만들기
- Intent Engine과 설명 레이어 연결하기
- 관리자 분석을 더 읽기 쉽게 다듬기
- 템플릿 라이브러리를 더 실용적으로 만들기

## 9. 한 줄 정리

PromptBuilder는 기능을 많이 넣는 방향이 아니라, 사용자가 덜 고민하고도 더 잘 쓸 수 있도록 구조를 정리해 온 프로젝트입니다.
