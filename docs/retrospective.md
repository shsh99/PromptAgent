# PromptBuilder Retrospective

이 문서는 PromptBuilder를 처음 다듬기 시작한 시점부터 현재까지의 고민, 실패, 개선, 그리고 남은 과제를 한 번에 정리한 회고 문서입니다.

## 왜 이 문서를 남기는가

- 기능은 계속 늘어나는데, 왜 그런 결정을 했는지 기록이 없으면 다음 수정이 어려워집니다.
- 테스트가 실패했을 때 어떤 원인으로 판단했고, 어떤 방식으로 고쳤는지 남겨두면 같은 실수를 줄일 수 있습니다.
- UI, 프롬프트 생성, 히스토리, 관리자 분석이 서로 얽혀 있기 때문에 작업 의도를 문서로 묶어둘 필요가 있습니다.

## 처음의 상태

초기에는 코드와 정적 자원이 한곳에 많이 몰려 있었습니다.

- `src` 루트에 엔트리와 기능 코드가 섞여 있었습니다.
- `public/static`에는 히스토리, Optimize, 프롬프트 관련 스크립트가 한 파일에 길게 붙어 있었습니다.
- 한글과 영어가 섞인 문구, 그리고 깨진 문자열이 곳곳에 남아 있었습니다.
- 화면은 기능은 많았지만, 글자 수에 비해 카드와 패널이 너무 넓어서 스크롤이 길었습니다.
- 히스토리는 이전 버전을 보기 어렵고, 관리자 모드도 진입 동선이 불안정했습니다.

이 시점의 핵심 문제는 "기능 부족"이 아니라 "구조와 표현의 혼잡함"이었습니다.

## 첫 번째 실패: 문자열과 경로 문제

가장 먼저 드러난 실패는 빌드 오류였습니다.

- `prompt-services.ts` 안의 깨진 문자열 때문에 Vite/esbuild가 파싱에 실패했습니다.
- 레이어 분리 이후에는 존재하지 않는 모듈에서 함수를 다시 꺼내오려 해서 export/import 오류가 반복됐습니다.
- `index.tsx`가 bootstrap 경로를 잘못 참조해 런타임 로딩 오류가 났습니다.

이때의 판단은 단순했습니다.

- "한 번에 크게 바꾸면 더 많은 곳이 깨진다."
- 그래서 먼저 빌드가 통과하도록 문자열과 import 경계를 정리했습니다.

## 두 번째 실패: 기능 분리 후 모듈 경계 혼선

기능별로 파일을 나누는 과정에서 다음 문제가 생겼습니다.

- `helpers.ts`, `helpers-config.ts`, `helpers-fields.ts`, `quality.ts`의 책임 경계가 헷갈렸습니다.
- 공통 헬퍼가 잘못된 파일을 참조해 export 오류가 발생했습니다.
- 분리 자체는 맞았지만, 분리 후 연결 규칙이 충분히 정리되지 않았습니다.

이때의 결론은 "분리보다 더 중요한 것은 경계 정의"였습니다.

그래서 다음 원칙을 세웠습니다.

- 설정 조회는 config 쪽으로 둔다.
- 생성 조합은 fields 쪽으로 둔다.
- 품질 분석은 quality 쪽에 둔다.
- helpers.ts는 얇은 연결층으로 유지한다.

## 세 번째 실패: 화면은 기능을 담았지만 읽기 어려웠다

UI는 기능을 모두 보여주려다 보니 세로로 너무 길어졌습니다.

- 템플릿 모드와 빌더 모드가 동시에 넓게 펼쳐졌습니다.
- 선택 입력 추가, 품질 요약, 결과 카드, 히스토리, 관리자, 업데이트가 모두 같은 비중으로 보였습니다.
- 라이트/다크 모드에 따라 가시성이 무너지는 영역이 생겼습니다.

여기서의 의사결정은 "기능 삭제가 아니라 밀도 조정"이었습니다.

- 카드 패딩과 섹션 간격을 줄였습니다.
- 공통 surface 토큰으로 표면 톤을 통일했습니다.
- 다크 모드와 라이트 모드의 예외 스타일을 분리했습니다.
- 템플릿, 선택 입력, 결과 카드, 관리자 패널의 가독성을 다시 맞췄습니다.

## 네 번째 실패: 히스토리가 최신 프롬프트만 보였다

히스토리 기능은 저장은 되었지만, 사용자가 이전 버전을 쉽게 보지 못했습니다.

- 세로 스크롤로 오래 내려가야 했습니다.
- 버전이 있어도 현재 선택 상태가 명확하지 않았습니다.
- 불러오기와 복사가 항상 최신만 기준이 되기 쉬웠습니다.

그래서 히스토리를 다음처럼 바꿨습니다.

- 카드 내부에 버전 탭을 추가했습니다.
- 선택된 버전을 기준으로 미리보기, 불러오기, 복사가 동작하도록 했습니다.
- 이전 버전을 찾는 동작을 스크롤보다 탭 전환 중심으로 바꿨습니다.

이 결정의 기준은 "기록을 보존하는 것"이 아니라 "기록을 빨리 꺼내는 것"이었습니다.

## 다섯 번째 실패: 프롬프트가 너무 한 덩어리였다

프롬프트 생성부는 오랫동안 다음이 섞여 있었습니다.

- 시스템 스타일
- 템플릿 구조
- 사용자 입력
- 최종 검증
- 전략 가이드

이 구조는 동작은 했지만 토큰이 많고, 역할이 섞여 있었습니다.

그래서 레이어를 분리했습니다.

- `System`은 스타일과 가드레일을 담당합니다.
- `Template`은 구조를 담당합니다.
- `User Input`은 키워드와 실제 입력을 담당합니다.

추가로 복잡도에 따라 두 가지 출력 경로를 뒀습니다.

- `low`: 간단 템플릿과 축약 검증
- `high`: 확장 템플릿과 전략 가이드 포함

이 변경의 목적은 다음 세 가지였습니다.

- 토큰 낭비를 줄인다.
- 응답 속도와 비용을 줄인다.
- 기능은 유지하면서 입력 난이도에 맞는 출력을 만든다.

## 현재까지의 개선 원칙

지금까지 적용한 개선의 공통 원칙은 거의 같습니다.

- 기능은 유지한다.
- 문구는 한국어/영어가 섞이지 않게 정리한다.
- 오류는 구조 문제로 보고 분리한다.
- UI는 간단해 보여야 하지만, 기능 유실은 없어야 한다.
- 테스트와 빌드를 통과시키면서 바꾼다.
- 바뀐 이유를 README, phase, plan, changelog에 같이 남긴다.

## 실제로 진행한 큰 변화

- 프롬프트 품질 분석기 강화
- 전략 가이드와 최종 검증 블록 추가
- 프롬프트 레이어 분리
- complexity 기반 출력 분기
- 홈 화면 밀도 조정
- surface 토큰 기반 UI 통일
- 템플릿 모드 복구
- 다크/라이트 가시성 개선
- 히스토리 버전 탭 전환
- 히스토리 선택 목록과 상세 패널을 분리해 세로 스크롤 부담을 줄임
- 관리자 모드 진입 및 로그 흐름 정리
- 문서와 changelog 동기화

## 작업하면서 얻은 결론

1. 파싱 에러는 구조보다 먼저 해결해야 한다.
2. 분리 작업은 파일을 나누는 것보다 책임을 명확히 하는 것이 더 중요하다.
3. UI는 예쁘게 넓히는 것보다, 읽기 쉬운 밀도로 다시 배치하는 것이 중요하다.
4. 히스토리는 저장보다 탐색이 중요하다.
5. 프롬프트는 길이보다 레이어와 목적이 중요하다.
6. 작업 기록은 기능만큼 중요하다.

## 남은 과제

- `public/static` 쪽 오래된 문자열과 로그는 아직 일부 정리가 남아 있습니다.
- Intent Engine, Compare, Admin Analytics는 더 정교한 UX가 필요합니다.
- 프롬프트 비교와 버전 비교는 현재보다 더 명확한 비교 뷰가 있으면 좋습니다.
- 문서의 한국어/영어 혼용도 계속 정리할 여지가 있습니다.

## 한 줄 요약

PromptBuilder는 "프롬프트를 만들어주는 도구"에서 "프롬프트를 더 잘 만들고, 더 쉽게 비교하고, 더 쉽게 다시 쓰는 도구"로 이동해 왔습니다.

이 변화는 기능 추가보다, 실패를 좁히고 구조를 정리하는 과정에서 이루어졌습니다.
## 회고 요약

이 문서는 처음의 오류 수정부터 현재의 구조 정리와 UI 개선까지, 우리가 어떤 문제를 발견했고 어떤 판단으로 바꿨는지를 남기기 위한 기록입니다.

### 처음에 보였던 문제

- Vite와 esbuild에서 문자열 오류가 나서 앱이 빌드되지 않았습니다.
- 한글과 영어, 깨진 문자열이 섞여 있어 화면과 문서를 읽기 어려웠습니다.
- 관리자 화면은 로딩이 멈추거나 버튼 글자가 안 보이는 문제가 있었습니다.
- 히스토리는 이전 버전을 찾기 위해 아래로 계속 스크롤해야 했습니다.
- 프롬프트 생성은 설정, 구조, 입력이 너무 섞여 있어 수정하기 어려웠습니다.

### 우리가 내린 판단

- 기능을 줄이기보다 구조를 나누는 편이 더 낫다고 봤습니다.
- 화면은 넓게 비우기보다 정보 위계를 정리하는 쪽으로 개선했습니다.
- 관리자와 히스토리는 스크롤 중심이 아니라 선택 중심으로 바꾸는 것이 맞다고 판단했습니다.
- 프롬프트는 System / Template / User Input 레이어로 나눠야 유지보수가 쉬워진다고 봤습니다.

### 실제로 바꾼 것

- 프롬프트 품질 분석과 생성 로직을 파일별로 나눴습니다.
- 히스토리를 카드 목록 + 상세 패널 + 버전 탭 구조로 바꿨습니다.
- 관리자 화면을 프로젝트에 맞는 카드형 대시보드로 다시 구성했습니다.
- 다크 모드와 라이트 모드의 토큰을 분리해서 버튼 가시성을 개선했습니다.
- 업데이트 로그와 phase / plan / README를 함께 최신화하는 규칙을 만들었습니다.

### 남은 방향

- 결과 3개 생성, Optimize, Compare/히스토리, Intent Engine, 관리자 분석 순서로 페이즈를 이어가야 합니다.
- 문서와 구현이 서로 어긋나지 않도록 변경 때마다 함께 기록해야 합니다.
- 가독성을 해치지 않는 선에서 밀도를 더 높이고, 스크롤은 더 줄여야 합니다.
