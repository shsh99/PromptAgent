# 워킹트리 / CLI 관리 가이드

이 문서는 PromptBuilder 작업을 여러 브랜치와 CLI 명령으로 관리할 때의 기준을 정리합니다.

## 목적

- 작업 충돌을 줄인다
- 리뷰 단위를 작게 만든다
- 배포 전 검증을 고정한다
- 신규 합류자의 인수인계를 쉽게 만든다

## 기본 원칙

1. 작업은 작게 쪼갠다
2. 한 번에 하나의 목적만 가진다
3. CLI 명령은 스크립트화한다
4. 결과는 로그와 문서로 남긴다
5. 배포 가능한 상태를 유지한다

## 브랜치 전략

- `main`: 배포 가능 상태
- `codex/<feature-name>`: 기능 작업
- `codex/docs/<topic>`: 문서 작업

작업 규칙:

- 큰 기능은 하나의 브랜치에 몰아넣지 않는다
- 코드와 문서 변경이 함께 필요하면 같은 작업 단위로 묶되, 커밋 메시지에서 분리 의도를 드러낸다
- 충돌 가능성이 높으면 워킹트리를 나눠서 작업한다

## 워킹트리 사용 시점

- 여러 기능을 병렬로 확인해야 할 때
- 리뷰용 브랜치와 개발용 브랜치를 분리하고 싶을 때
- 배포 전 상태를 따로 보존하고 싶을 때

권장 확인 항목:

- 현재 브랜치 이름
- 워킹트리 경로
- 작업 대상 파일 목록
- 병합 예정 브랜치

## CLI 운영 규칙

### 개발

- `npm run dev`
- `npm run build`
- `npm run test`

### 배포 전

- `git status`
- `git diff --stat`
- `npm run build`
- 필요 시 `npm run test`

### 배포 후

- 배포 결과 확인
- 변경 로그 반영
- 문서 업데이트

## 체크리스트

- [ ] 브랜치 목적이 명확하다
- [ ] 작업 범위가 작다
- [ ] 배포 전 검증 명령을 실행했다
- [ ] 변경 로그를 남겼다
- [ ] PR/MR 템플릿을 채웠다
- [ ] 문서가 동기화됐다

## 포트폴리오에 넣을 때의 메시지

이 가이드는 단순한 개발 습관이 아니라, 팀 단위 운영을 염두에 둔 구조라는 점을 보여줍니다.  
특히 MZC나 MSP 관점에서는 기능보다도 "누가 와도 유지할 수 있는가"가 더 중요하므로, 워킹트리와 CLI 운영 규칙을 문서로 남기는 것이 설득력 있습니다.
