# PromptBuilder 포트폴리오

> 프롬프트를 "잘 쓰는 법"이 아니라 "운영 가능한 작업 단위"로 바꾸는 워크스페이스

## 프로젝트 개요

- 프로젝트명: PromptBuilder
- 프로젝트 유형: 프롬프트 품질 분석 및 개선 워크플로우 도구
- 배포 링크: https://promptbuilder-df6.pages.dev/
- 핵심 가치: 구조화, 재현성, 버전 관리, 비교, 운영성

PromptBuilder는 단순 생성기가 아닙니다.  
프롬프트를 입력받아 구조를 분석하고, 결과를 평가하고, 다음 버전을 제안하고, 히스토리로 남기는 흐름 전체를 다루는 도구입니다.

이 포트폴리오는 기능 목록보다도, 이 프로젝트가 "서비스로서 왜 신뢰할 수 있는지"를 보여주는 데 초점을 둡니다.

## 이 포트폴리오가 겨냥하는 평가 기준

이 프로젝트는 다음 질문에 답할 수 있어야 합니다.

- 이 서비스는 특정 CSP에 과도하게 묶여 있지 않은가
- 인프라가 서비스 목적에 비해 과하게 복잡하지 않은가
- 운영 인수인계가 가능한가
- PR, MR, 체크리스트, 코드 리뷰가 실제로 작동하는가
- AI를 썼다면 어디에, 어떤 기준으로 썼는가
- 네이밍과 테이블 설계가 유지보수 관점에서 납득 가능한가
- 데모와 운영 환경이 일관되게 설명되는가

즉, "무엇을 만들었는가"보다 "어떻게 운영 가능한 형태로 만들었는가"를 보여줘야 합니다.

## 문제 정의

프롬프트 작업은 생각보다 체계가 필요합니다.

- 시작점이 불명확해서 초안 작성 비용이 크다
- 역할, 목표, 입력, 제약, 출력 형식이 쉽게 누락된다
- 개선 기준이 없어서 같은 실수를 반복한다
- 모델이 달라지면 같은 문장도 결과가 달라진다
- 결과를 버전으로 남기지 않으면 재현이 어렵다

PromptBuilder는 이 문제를 "문장 생성"이 아니라 "작업 계약 설계" 문제로 봤습니다.

## 해결 방향

핵심 설계는 다음 원칙을 따릅니다.

- 프롬프트를 역할, 문제 정의, 입력 데이터, 목표, 제약, 출력 형식으로 분리
- 품질 분석을 통해 누락과 모호함을 먼저 드러냄
- 모델별 최적화를 통해 문장 구조를 재작성
- 결과를 버전으로 저장하고 비교 가능하게 유지
- 히스토리와 템플릿으로 재사용성을 확보

이 방식은 "멋진 AI 데모"보다 "계속 운영할 수 있는 도구"에 가깝습니다.

## 서비스 평가에 대한 선제 설명

이 포트폴리오는 다음 우려를 미리 해소하는 방향으로 작성되어야 합니다.

### 1. CSP 종속성

- 핵심 도메인 로직은 프롬프트 분석, 버전 관리, 기록 구조입니다
- Cloudflare는 배포와 런타임 선택지일 뿐, 제품의 본질은 CSP 특화 기능이 아닙니다
- 외부 서비스 변경이 있어도 프롬프트 구조와 평가 흐름은 유지되도록 설계해야 합니다

### 2. 인프라 복잡도

- 서비스 목적은 프롬프트 작업 효율화이지, 인프라 과시가 아닙니다
- 최소한의 구성으로 운영 가능해야 하며, 복잡한 네트워크 구성은 명확한 이유가 있어야 합니다
- "기술을 많이 썼다"보다 "운영 비용과 복잡도를 줄였다"가 설득력 있어야 합니다

### 3. 운영 인수인계

- 업무가 여러 사람에게 넘어갈 수 있다는 전제를 두고 문서와 체크리스트를 남겨야 합니다
- 기능별 PR 템플릿, 리뷰 기준, 회귀 확인 항목이 있어야 합니다
- 신규 인원도 구조를 따라가면 빠르게 적응할 수 있어야 합니다

### 4. 데모와 운영의 일관성

- 데모는 예쁘기만 하면 안 되고, 운영과 연결되어야 합니다
- 포트폴리오에 적는 배포 환경, 저장소, 운영 방식이 서로 충돌하면 신뢰가 무너집니다
- "어디서 돌았는지"와 "어떻게 관리했는지"를 함께 설명해야 합니다

### 5. AI 활용 방식

- AI는 리뷰어가 아니라 도구입니다
- 어디까지 자동화했고, 어디부터 사람이 판단했는지 경계를 분명히 해야 합니다
- 결과 검증 기준이 없으면 AI 사용 자체가 오히려 신뢰를 떨어뜨립니다

## 주요 기능

### 1. 프롬프트 품질 분석

- 프롬프트의 모호한 부분과 누락된 요소를 점검
- 출력 형식, 입력 구조, 제약 조건, 목표 적합성을 함께 확인
- 개선 포인트를 체크리스트 형태로 정리

### 2. 모델별 최적화

- GPT, Claude, Gemini 성향에 맞춰 지시 문장을 재작성
- 구조 중심, 문맥 중심, 직접 지시형 등 스타일을 분리
- 단일 원문을 여러 모델에 맞는 실행형 프롬프트로 변환

### 3. 결과 3안 생성

- 하나의 입력으로 여러 결과 버전을 생성
- 후보를 비교해서 더 나은 방향을 고르는 구조
- 단일 정답보다 검토 가능한 후보군을 우선

### 4. 버전 관리와 비교

- 의미 있는 수정마다 버전을 저장
- 이전 버전과의 차이를 확인하고 되돌릴 수 있음
- 수정 이유를 남겨 다음 개선에 활용

### 5. 히스토리와 라이브러리

- 작성한 프롬프트와 결과를 히스토리로 누적
- 반복 작업을 위한 재사용 템플릿 제공
- 예시와 보조 문구로 진입 장벽을 낮춤

### 6. 관리자 분석 화면

- 사용 로그와 활동 로그를 관리자가 확인
- 카드형 대시보드로 핵심 지표를 빠르게 파악
- 긴 스크롤보다 탭 중심의 탐색 구조를 우선

## 기술 스택

- Frontend: TypeScript, Vite
- Backend: Hono
- Runtime / Hosting: Cloudflare Pages, Cloudflare Workers
- Database: Cloudflare D1
- Tooling: Wrangler, Node.js

## 시스템 구조

- `webapp/src/app/routes.ts`
  - API 라우팅
  - 로그 저장
  - 개선 흐름 연결
- `webapp/src/features/prompt/*`
  - 품질 분석
  - 필드 생성
  - 프롬프트 생성과 최적화
- `webapp/public/static/features/*`
  - 실제 UI 동작
  - 히스토리, 최적화, 관리자 기능
- `docs/*`
  - 제품 원칙
  - 단계별 계획
  - 기능 스펙
  - 회고

## 협업과 운영 방식

이 프로젝트를 포트폴리오로 보여줄 때는 기능보다 운영 방식이 더 중요합니다.

- PR/MR 템플릿을 두고 기능별 체크리스트를 남김
- 실제 템플릿은 `.github/pull_request_template.md` 같은 위치에 두는 것이 좋음
- 코드 리뷰와 AI 리뷰의 역할을 분리
- 브랜치, 워킹트리, 로컬 작업 단위를 명시
- 신규 합류자가 따라올 수 있는 문서 흐름을 유지
- 리뷰 결과를 작업 항목으로 되돌리는 루프를 만들기

이 항목이 빠지면 "개발은 했지만 운영은 안 보이는 프로젝트"로 읽힙니다.

## 문서화 원칙

포트폴리오에는 단순 결과만 쓰지 않고, 다음 내용을 함께 보여주는 편이 좋습니다.

- 무엇을 만들었는가
- 왜 그렇게 설계했는가
- 어떤 제약을 받아들였는가
- 무엇을 운영 기준으로 삼았는가
- 인수인계는 어떻게 할 것인가
- 리뷰와 회귀 방지는 어떻게 할 것인가

이 프로젝트는 특히 문서가 중요합니다.  
기능 설명만 있으면 작은 데모처럼 보이지만, 운영 기준과 리뷰 기준까지 들어가면 팀 단위 프로젝트로 읽힙니다.

## 내가 맡은 역할

- 프롬프트 하네스 구조 설계
- 품질 분석 기준 정리
- 모델별 최적화 흐름 구현
- 버전 저장과 비교 구조 설계
- 관리자 분석 화면과 로그 구조 설계
- 기능 문서와 회고 문서의 동기화
- 운영과 협업을 고려한 체크리스트 정리

## 포트폴리오에 꼭 들어가야 하는 보완 항목

지금처럼 평가받지 않으려면 아래 항목이 포트폴리오에 명시되어야 합니다.

- PR/MR 템플릿 예시
- 기능 체크리스트
- 코드 리뷰 기준
- AI 보조 사용 범위
- 워킹트리 또는 브랜치 관리 방식
- 배포 환경과 데모 환경의 차이
- 외부 의존성과 대체 가능성
- 테이블 설계와 네이밍 의도
- 운영 인수인계 문서

## 성과와 의미

PromptBuilder의 의미는 단순히 프롬프트를 생성하는 데 있지 않습니다.

- 프롬프트 작성의 시작점을 구조화했다
- 결과를 보고 다시 개선하는 루프를 만들었다
- 단순 생성보다 진단과 재사용을 강조했다
- 사용자 기능과 운영 기능을 함께 다루는 관점을 제시했다

정량 수치는 별도 표기하되, 포트폴리오의 핵심 메시지는 다음이어야 합니다.

> 이 프로젝트는 AI를 붙인 데모가 아니라, 운영 가능한 프롬프트 작업 플랫폼이다.

## 향후 개선 포인트

- 결과 비교 UX를 더 직관적으로 정리
- Intent Engine과 설명 레이어 강화
- 초보자용 예시와 가이드 보강
- 템플릿 라이브러리 확장
- 공유 및 협업 기능 추가
- PR 템플릿과 리뷰 체크리스트 문서화
- 운영 인수인계 절차 보강

## 한 줄 요약

PromptBuilder는 프롬프트를 단순 작성물이 아니라 개선 가능한 작업 단위로 바꾸고, 품질 분석부터 버전 비교, 운영 문서화까지 이어주는 구조화된 프롬프트 워크스페이스입니다.
