# PromptBuilder 포트폴리오 리뷰 팩

이 문서는 포트폴리오를 평가받을 때 자주 나오는 질문에 먼저 답하기 위한 보완 문서입니다.

## 1. 가져올 만한 기능

참고 프로젝트 `promptbuilder-pro-complete`에서 가져올 가치가 있는 기능은 아래와 같습니다.

| 기능 | 가져올 가치 | 현재 프로젝트 반영 포인트 |
| --- | --- | --- |
| 스마트 입력 / 의도 분류 | 초보자 진입 장벽을 낮춤 | 자연어 입력을 구조화된 작업으로 변환 |
| 목적 -> 방식 -> 필드 위저드 | 흐름이 명확하고 시연이 쉬움 | 단계형 UX 또는 가이드형 입력으로 정리 |
| 템플릿 추천 | 재사용성과 표준화가 보임 | 자주 쓰는 프롬프트 패턴 라이브러리화 |
| 품질 점수 / 체크리스트 | 평가 가능한 제품처럼 보임 | 점검 항목을 명시하고 개선 사유를 남김 |
| 버전 저장 / 비교 | 개선 루프가 드러남 | 히스토리, diff, rollback 구조 강조 |
| 모델 변환 | 모델별 최적화 역량을 보여줌 | GPT, Claude, Gemini 별 rewrite 분리 |
| A/B 테스트 | 실험 가능한 제품으로 보임 | 결과 비교와 채택 근거를 남김 |
| 복사 / 다운로드 / 저장 | 실사용 기능으로 신뢰도 상승 | 결과 export와 library 저장 제공 |

### 우선순위

1. 품질 체크리스트
2. 버전 비교
3. 템플릿 라이브러리
4. 의도 분류
5. 모델별 최적화

이 순서가 좋은 이유는, 단순히 "기능이 많다"가 아니라 "개선과 운영이 가능하다"는 인상을 먼저 주기 때문입니다.

## 2. 포트폴리오에 넣을 문장

아래 문장은 서비스 소개보다 신뢰도와 운영성을 먼저 보여주도록 다듬은 버전입니다.

### 짧은 소개

PromptBuilder는 프롬프트를 단순 작성물이 아니라, 분석과 개선, 버전 관리가 가능한 작업 단위로 바꾸는 구조화된 워크스페이스입니다.

### 평가 방어형 소개

이 프로젝트는 프롬프트 생성만 하는 도구가 아니라, 의도 분류, 품질 점검, 모델별 최적화, 버전 비교, 히스토리 저장까지 이어지는 운영 가능한 흐름을 제공하는 것이 핵심입니다.  
따라서 포트폴리오에서는 기능의 화려함보다, 재현성, 인수인계 가능성, 리뷰 기준, 운영 구조를 함께 보여주는 것이 중요합니다.

### 엔터프라이즈 톤 소개

PromptBuilder는 프롬프트 작업의 진입 장벽을 낮추는 동시에, 품질 기준과 변경 이력을 명시적으로 관리할 수 있게 설계된 도구입니다.  
기능 자체보다도, 협업과 운영에서 요구되는 체크리스트, 리뷰 흐름, 버전 관리, 문서 동기화를 함께 고려한 점이 특징입니다.

## 3. 운영 체계

포트폴리오에서 가장 중요하게 보이는 부분은 "누가 봐도 운영 가능한가"입니다.

### 3-1. 워킹트리 관리

- 기능 단위로 작업 브랜치를 분리합니다.
- 같은 작업을 여러 사람이 만질 때는 워킹트리 또는 명시적 브랜치로 충돌을 줄입니다.
- 작업 시작 전에 현재 기준 브랜치와 작업 범위를 문서화합니다.
- 작업 종료 후에는 병합 대상과 미반영 변경을 명확히 적습니다.

권장 방식:

1. `main`은 항상 배포 가능한 상태로 유지
2. 기능 작업은 `codex/<feature-name>` 형태의 브랜치 사용
3. 큰 작업은 워킹트리로 분리해서 동시에 검토 가능하게 유지
4. 작업 전후로 `git status`, `git diff --stat`, `git branch --show-current`를 확인

### 3-2. CLI 관리

- CLI 작업은 목적별로 나눕니다.
- 빌드, 테스트, 배포, 로그 확인을 한 군데서 섞지 않습니다.
- 반복 명령은 스크립트로 고정해서 사람마다 다른 실행을 줄입니다.
- 배포 전에 항상 로컬 검증 명령을 먼저 수행합니다.

권장 체크:

1. `npm run build`로 산출물 확인
2. `npm run test`로 회귀 확인
3. `git diff`로 변경 범위 확인
4. 배포 전 `README`, `docs`, 템플릿 동기화 확인

### 3-3. PR / MR 템플릿

포트폴리오에는 아래 항목이 실제 템플릿으로 존재해야 설득력이 생깁니다.

- 변경 요약
- 기능 체크리스트
- 검증 결과
- 운영 메모
- AI 보조 사용 범위
- 추가 리뷰 필요 항목

이미 적용한 템플릿:

- [`.github/pull_request_template.md`](../.github/pull_request_template.md)

### 3-4. 리뷰 흐름

1. 기능 의도 확인
2. 체크리스트 검증
3. 코드 리뷰
4. 필요 시 AI 보조 리뷰
5. 회귀 테스트
6. 문서 동기화
7. 병합

이 흐름이 있어야 "개발은 했지만 운영은 안 보이는 상태"를 피할 수 있습니다.

### 3-5. 문서 체계

포트폴리오와 개발 문서는 아래처럼 분리하면 좋습니다.

- `README.md`: 외부 설명
- `docs/portfolio.md`: 포트폴리오 본문
- `docs/portfolio-review-pack.md`: 평가 방어용 요약
- `docs/working-tree-cli-guide.md`: 작업 운영 방식
- `docs/README.md`: 문서 인덱스

## 결론

이 프로젝트에서 가장 강한 메시지는 "AI를 쓴 서비스"가 아니라 "운영 가능한 구조를 갖춘 서비스"라는 점입니다.  
그래서 포트폴리오에는 기능보다도 평가 기준, 운영 체계, 리뷰 방식, 문서화 규칙을 함께 넣는 편이 훨씬 안전합니다.
